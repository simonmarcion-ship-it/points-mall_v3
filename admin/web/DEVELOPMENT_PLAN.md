# 积分商城后台开发计划

## 1. 当前状态

当前项目已经完成了第一轮工程拆分，但还不能直接交付甲方生产使用。

现有结构：

```text
积分商城/
  crawler/              旧商城爬虫脚本与已爬取 Excel 数据
  web/
    backend/            FastAPI 后端
    frontend/           静态前端页面
    data/mall.db        SQLite 数据库
    run_server.py       本地启动入口
    README.md           当前说明文档
```

已具备能力：

- 从 `crawler` 目录下的 Excel 导入客户、优惠券、券模板数据。
- 使用 SQLite 持久化客户、券、券模板、操作流水。
- 提供客户查询、客户详情、发券、核销、券模板查看、操作记录查看页面。
- 发券和核销已经通过后端接口写入数据库，不再只存在内存或 JSON 文件中。

当前不足：

- 没有登录、账号、角色和权限控制。
- 没有正式部署方式。
- 没有生产数据库选型和备份方案。
- 没有完整的数据导入、对账、回滚流程。
- 核销逻辑还需要补并发保护、过期判断、适用规则判断。
- 前端还是轻量静态页面，适合内部验证流程，不适合直接作为长期可维护的甲方系统。
- 缺少测试、日志、配置管理、异常监控。

## 2. 最终交付目标

目标是建设一个公司内部人员使用的积分商城后台系统，用于承接旧微盟商城月底到期前迁出的客户和优惠券数据。

第一版生产系统应支持：

- 员工登录后台。
- 按手机号、客户编号、昵称查询客户。
- 查看客户基础信息、积分、余额、优惠券明细。
- 给客户发券。
- 核销客户优惠券。
- 管理券模板。
- 查询发券、核销、作废、导入等操作记录。
- 从旧商城爬取结果导入客户和优惠券数据。
- 对导入数据做数量对账。
- 提供数据库备份和恢复方案。

生产交付必须满足：

- 数据不能丢。
- 券不能重复核销。
- 所有关键操作必须可追溯。
- 普通员工不能越权操作。
- 系统异常时能定位原因。
- 旧商城最终迁移时有明确流程和回退方案。

## 3. 阶段一：本地 MVP 完善

目标：把当前原型完善成一个功能闭环清楚、可以内部试用的版本。

### 3.1 数据库结构完善

任务：

- 明确 `customers`、`coupon_templates`、`coupons`、`operation_logs` 字段含义。
- 为券增加更完整的状态字段：`unused`、`used`、`expired`、`voided`。
- 为券增加来源字段：`old_mall`、`manual`、`import`。
- 为发券和核销增加操作人字段。
- 为客户增加备注字段和标签字段，方便后期人工维护。

验收标准：

- 数据库表结构文档化。
- 数据能从 Excel 重复导入到空库。
- 导入后客户数、券数、模板数可以被统计。
- 手动发券和核销后，数据库中有完整记录。

### 3.2 核销逻辑补强

任务：

- 核销时必须检查券状态是否为 `unused`。
- 核销时必须检查券是否过期。
- 核销时必须使用数据库事务更新状态。
- 核销成功后写入操作流水。
- 重复核销同一券码时必须失败。

验收标准：

- 未使用券可以核销。
- 已核销券再次核销会被拒绝。
- 已过期券不能核销。
- 不存在的券码会提示明确错误。
- 操作流水能查到核销人、时间、券码、客户。

### 3.3 发券逻辑补强

任务：

- 发券时检查客户是否存在。
- 发券时检查券模板是否启用。
- 支持一次发多张券。
- 自动生成唯一券码。
- 发券后写入操作流水。
- 支持备注。

验收标准：

- 给存在客户发券成功。
- 给不存在客户发券失败。
- 停用模板不能发券。
- 批量发券数量正确。
- 每张券码唯一。

### 3.4 前端页面补齐

任务：

- 客户列表增加分页。
- 客户详情页增加发券入口。
- 核销页增加核销前预览。
- 券模板页增加启用、停用、创建、编辑能力。
- 操作记录页增加筛选条件。
- 增加明确的成功和失败提示。

验收标准：

- 员工可以不看数据库完成客户查询、发券、核销。
- 常见错误都有前端提示。
- 页面在普通笔记本浏览器上能稳定使用。

## 4. 阶段二：账号、权限和审计

目标：让系统具备给公司内部多人使用的基本安全边界。

### 4.1 用户账号

任务：

- 增加 `users` 表。
- 支持用户名和密码登录。
- 密码必须哈希存储，不能明文保存。
- 登录后使用 session 或 token 保持状态。
- 支持退出登录。

验收标准：

- 未登录不能访问后台页面和 API。
- 登录成功后能正常使用系统。
- 密码在数据库中不是明文。

### 4.2 角色权限

建议角色：

- `admin`：管理员，可以管理用户、模板、导入数据、查看所有记录。
- `operator`：普通员工，可以查客户、发券、核销。
- `viewer`：只读用户，只能查询客户和券。

任务：

- 增加角色字段。
- 为 API 增加权限检查。
- 前端按权限隐藏不可用功能。

验收标准：

- 只读用户不能发券和核销。
- 普通员工不能管理用户。
- 管理员可以管理模板和账号。

### 4.3 审计日志

任务：

- 所有关键动作写入 `operation_logs`。
- 记录操作人、时间、动作、目标对象、请求结果、备注。
- 登录失败、登录成功也应记录。

验收标准：

- 能追溯谁在什么时间给哪个客户发了什么券。
- 能追溯谁在什么时间核销了哪张券。
- 能导出操作记录。

## 5. 阶段三：正式数据导入和对账

目标：为月底旧商城到期前的最终迁移做准备。

### 5.1 爬虫结果标准化

任务：

- 固定爬虫输出文件名和字段。
- 保留原始 Excel 或 JSON 作为不可修改备份。
- 每次爬取生成一个批次目录。

建议目录：

```text
crawler/output_batches/
  2026-05-31_final/
    customers.xlsx
    coupon_details.xlsx
    raw/
```

验收标准：

- 每次爬取结果都能独立保存。
- 不会覆盖历史爬取结果。
- 能知道某条数据来自哪个批次。

### 5.2 导入批次管理

任务：

- 增加 `import_batches` 表。
- 每次导入记录批次号、来源文件、导入时间、导入人。
- 记录导入客户数、券数、失败数。
- 支持空库导入。
- 支持导入前校验。

验收标准：

- 可以看到每次导入批次。
- 导入失败时能知道失败原因。
- 导入成功后能看到数量汇总。

### 5.3 对账

任务：

- 对比旧商城爬虫输出和新数据库数量。
- 对账维度包括客户数、总券数、未使用券数、已使用券数、过期券数。
- 输出对账报告。

验收标准：

- 导入前后数量一致或差异有解释。
- 对账报告可以保存。
- 差异数据可以定位到具体客户或券码。

## 6. 阶段四：生产数据库和部署

目标：把系统从本地 SQLite 版本升级到稳定可部署版本。

### 6.1 数据库选型

短期可以继续 SQLite，但真正给甲方用建议升级到 PostgreSQL 或 MySQL。

建议：

- 如果部署在单台 Windows 电脑、访问人数少，可以短期使用 SQLite。
- 如果多人同时使用、长期运行，建议使用 PostgreSQL。
- 如果甲方已有 MySQL 运维经验，也可以选择 MySQL。

验收标准：

- 数据库文件或服务位置明确。
- 有备份方式。
- 有恢复测试。

### 6.2 配置管理

任务：

- 使用 `.env` 管理数据库地址、管理员账号初始化信息、服务端口。
- 不把敏感配置写死在代码中。
- 区分开发环境和生产环境。

验收标准：

- 修改配置不需要改代码。
- 生产配置不会提交到代码仓库。

### 6.3 部署方式

可选方案：

- 单机部署：Windows 电脑运行 Python 服务，适合小团队内网使用。
- 服务器部署：Linux 服务器 + Nginx + Uvicorn/Gunicorn。
- Docker 部署：后端、数据库、备份任务容器化。

建议第一版：

- 如果甲方只是门店或办公室内部使用，先做单机或内网服务器部署。
- 等业务稳定后再容器化。

验收标准：

- 服务开机后能自动启动。
- 异常退出后能重启。
- 能通过固定地址访问。
- 有部署文档。

## 7. 阶段五：测试和验收

目标：在交给甲方前，证明系统关键流程可靠。

### 7.1 自动化测试

任务：

- 客户查询 API 测试。
- 发券 API 测试。
- 核销 API 测试。
- 重复核销测试。
- 权限测试。
- 导入测试。

验收标准：

- 核心接口测试可以一键运行。
- 发版前测试通过。

### 7.2 人工验收场景

必须覆盖：

- 员工登录。
- 搜索客户。
- 查看客户券。
- 给客户发券。
- 核销一张未使用券。
- 重复核销被拒绝。
- 使用无效券码被拒绝。
- 管理员查看操作记录。
- 管理员导入一批旧商城数据。

验收标准：

- 每个场景有明确通过或失败结果。
- 发现的问题有记录和修复确认。

## 8. 月底最终迁移流程

建议流程：

1. 确认旧商城停止新增发券和核销，或约定冻结时间点。
2. 执行最终全量爬取。
3. 保存爬虫原始输出，作为备份。
4. 在测试库导入最终数据。
5. 生成对账报告。
6. 业务人员抽查客户和券。
7. 备份生产数据库。
8. 清空或准备生产库。
9. 导入最终数据到生产库。
10. 再次对账。
11. 切换员工使用新后台。
12. 旧商城保留只读截图、Excel 和原始数据备份。

回退方案：

- 如果导入失败，停止切换，继续使用旧商城。
- 如果切换后发现严重问题，恢复切换前数据库备份。
- 所有最终迁移文件不得覆盖，必须保留。

## 9. 优先级建议

近期优先顺序：

1. 登录和权限。
2. 核销并发和过期判断。
3. 券模板增删改和停用。
4. 导入批次和对账。
5. 前端页面完善。
6. 部署和备份。
7. 自动化测试。

不要优先做的事情：

- 不要先做复杂营销活动。
- 不要先做客户移动端商城。
- 不要过早做大量视觉美化。
- 不要在数据迁移流程没有稳定前做大规模重构。

## 11. 推荐下一步

下一步建议先做登录和权限，因为这是从内部原型走向真实系统的第一道门槛。做完后，再补核销事务和过期判断，随后完善导入批次和对账流程。
